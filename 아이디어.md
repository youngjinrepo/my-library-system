시나리오

1. [동시성 제어] "해리포터 마지막 1권을 100명이 동시에 빌리려고 한다면?" (가장 추천 ⭐️)

상황: 인기 도서의 대출 가능 상태(SHELVED)를 여러 사용자가 동시에 LOANED로 변경하려고 할 때 발생하는 '경쟁 상태(Race Condition)'를 막아야 합니다. 아무런 처리를 하지 않으면 1권의 책이 2명에게 대출되는 초유의 사태가 벌어집니다.

다뤄볼 기술: JPA의 낙관적 락(Optimistic Lock), 비관적 락(Pessimistic Lock)을 코드로 구현해 보고, 더 나아가 Redis를 활용한 분산 락(Distributed Lock)까지 확장해 보는 방법을 심도 있게 다룹니다.

2. [이벤트 기반 비동기 처리] "대출 완료 시 알림톡 전송이 실패하면 대출도 취소되어야 할까?"

상황: 대출(Loan)이 성공하면 회원에게 카카오톡 알림을 보내야 합니다. 그런데 외부 알림 서버가 느려지면 대출 처리 시간도 같이 느려지고, 심지어 알림 전송이 실패했다고 애써 처리한 대출 DB까지 롤백(Rollback)되는 강결합 문제가 생깁니다.

다뤄볼 기술: 도메인 로직(대출)과 부가 로직(알림)을 완벽하게 분리하는 Spring Event (@TransactionalEventListener)와 비동기(@Async) 처리를 통해 시스템의 결합도를 낮추는 우아한 구조를 설계합니다.

3. [대규모 읽기 성능 최적화] "매번 DB를 찌르는 인기 도서 목록, 어떻게 개선할까?"

상황: 대출 기능보다 훨씬 많이 호출되는 것이 바로 '도서 검색'이나 '인기 도서 조회'입니다. 모든 사용자의 조회 요청이 DB로 향한다면 DB 부하가 극심해집니다.

다뤄볼 기술: 앞서 가볍게 언급했던 Spring Cache를 활용하여 본격적인 글로벌 캐싱 환경을 구축하고, 누군가 책을 대출해서 상태가 바뀌었을 때 캐시를 어떻게 깔끔하고 안전하게 지울지(Cache Eviction 전략) 고민해 봅니다.
